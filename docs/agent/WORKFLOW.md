# 智能体工作流

本文定义 natflow 仓库如何实践“代码仓库就是智能体记忆”。

## 核心模型

仓库记忆分为四层：

| 层级 | 内容 | 作用 |
| --- | --- | --- |
| 事实层 | 源码、Makefile、DKMS 配置 | 当前真实行为。 |
| 规格层 | `SYSTEM_DESIGN_SPEC.md`、`README.md` | 压缩实现、接口和使用约束。 |
| 协作层 | `AGENTS.md`、`docs/agent/MEMORY.md`、本文件 | 告诉智能体如何进入仓库和怎么工作。 |
| 过程层 | `docs/agent/DECISIONS.md`、任务模板、提交记录 | 保留为什么这样做，以及下一步怎么接。 |

智能体不是把记忆放在外部聊天记录里，而是把可复用、可审查、可版本化的记忆写回仓库。

## 每轮任务闭环

1. 读入口：`AGENTS.md` 和 `docs/agent/MEMORY.md`。
2. 定边界：确认任务影响代码、接口、文档、构建还是流程。
3. 查事实：读相关源码，不只依赖记忆文件。
4. 做改动：保持范围小，遵循现有风格。
5. 跑验证：按风险选择 `git diff --check`、`make` 或更具体的测试。
6. 写回记忆：只记录对后续有价值的长期事实。
7. 交接：说明改动、验证结果和残留风险。

## 功能开发

功能类任务建议按这个顺序展开：

1. 在 `SYSTEM_DESIGN_SPEC.md` 找到相关模块和接口。
2. 阅读对应 `natflow_*.c/.h` 的真实实现。
3. 明确是否会改变字符设备协议、sysctl、状态值、编译宏或数据面行为。
4. 先实现最小行为，再更新文档和记忆。
5. 如果存在兼容性破坏，必须在 `docs/agent/DECISIONS.md` 记录原因。

## Bug 修复

Bug 修复任务要优先形成可验证假设：

1. 复述现象和预期行为。
2. 找到最小相关路径，例如 path、user、zone、urllogger 或 conntrack dump。
3. 查清锁、RCU、skb 生命周期、net_device 生命周期和错误返回。
4. 修复后补充回归说明或验证命令。
5. 如果 bug 暴露了规格缺口，更新 `SYSTEM_DESIGN_SPEC.md` 或本记忆层。

## 文档更新

文档任务也要回到源码确认：

1. 用户命令、输出格式、默认值以源码为准。
2. `README.md` 面向部署和对接人员，写使用方式和限制。
3. `SYSTEM_DESIGN_SPEC.md` 面向开发和审查，写内部流程和兼容约束。
4. `docs/agent/` 面向智能体，写工作方法、索引和长期判断。

## 任务交接

复杂任务完成后，在最终说明中至少包含：

- 修改了哪些文件。
- 行为或接口是否改变。
- 执行了哪些验证。
- 未验证或仍有风险的部分。
- 需要下一轮继续的具体事项。

如果任务需要暂停，可以把交接信息写进任务文档或提交说明，避免下一轮只依赖聊天历史。

## 何时不写入长期记忆

以下内容不要写入长期记忆：

- 一次性命令输出。
- 未经源码确认的猜测。
- 临时调试路径。
- 与仓库无关的通用 AI 提示词。
- 已经能从小段源码直接看出的局部实现细节。

长期记忆的目标是减少后续智能体的认知成本，而不是复制整个仓库。
